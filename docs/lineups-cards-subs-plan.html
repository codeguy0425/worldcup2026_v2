Lineups, Cards & Substitutions Integration Plan — Phase 1 · WC2026 Redesign

This document analyses the current data landscape and lays out a phased approach for displaying starting XIs, bookings, and substitutions on the match page and beyond. No code has been changed — this is a planning document only.

2. Proposed Data Model

Extend the Match interface with three new optional arrays. All new fields sit inline in matches.json — the match page already fetches it, so no extra useJson calls needed.

// ——— New interfaces ———

interface TeamLineup {
  teamId: string
  formation?: string         // e.g. "4-3-3", "3-4-2-1"
  startingXI:  PlayerInLineup[]  // 11 starters
  substitutes: PlayerInLineup[]  // bench (up to 15)
}

interface PlayerInLineup {
  no:       number   // shirt number
  pos:      string   // GK / DF / MF / FW or specific (CB, CM, CF…)
  name:     string
  captain?: boolean
}

interface BookingEvent {
  minute:        number
  stoppageTime?: number
  teamId:        string
  player:        string
  card:          'yellow' | 'red' | 'second-yellow'
}

interface SubEvent {
  minute:        number
  stoppageTime?: number
  teamId:        string
  off:  { no: number; name: string }
  on:   { no: number; name: string }
}

These get added to the existing Match interface as optional fields:

interface Match {
  // … existing fields …
  goals?:          GoalEvent[]
  lineups?:        TeamLineup[]
  bookings?:       BookingEvent[]
  substitutions?:  SubEvent[]
}
Card detail limitation: The Wikipedia statistics section only provides aggregate counts per team (yellow: 0, red: 0). Per-player booking data requires scraping FIFA's tactical PDFs. The plan supports both — aggregate cards can be shown as team totals, per-player as detailed events when available.

Recommended Start: Wikipedia Scraper + Manual Override

Begin with a Wikipedia-based enrichment script (similar to enrich-goals.mjs), covering the matches that do have articles. Use a manual override file for the rest. The PDF parser can be added later when more coverage is needed.

Wikipedia article URL pattern: https://en.wikipedia.org/wiki/{Team1}_v_{Team2}_({year}_FIFA_World_Cup)
The scraper tries this convention and falls back gracefully when the article doesn't exist (HTTP 404).

Scraper pseudo-flow

for each match M in matches:
  if M has no score → skip (not played yet)
  article = fetch(PAGE_URL(M.team1, M.team2))
  if article is 404 → continue (no data available)

  // Parse the "Details" section (section index 9)
  details = article.getSection("Details")

  // Extract starting XI + bench for each team
  // Wikipedia HTML uses rows of: pos | shirt | player(sub-icon)(c)
  M.lineups = extractLineups(details, M.team1Id, M.team2Id)

  // Extract substitutions ("Substitutions:" row)
  M.substitutions = extractSubstitutions(details, M.team1Id, M.team2Id)

  // Extract aggregate card counts from the "Statistics" section
  stats = article.getSection("Statistics")
  M.bookings = extractCardCounts(stats, M.team1Id, M.team2Id)

  // Patch matches.json in-place (same pattern as enrich-goals.mjs)

Pipeline integration

In update.mjs, the cleanMatches function needs to pass through the new fields:

const entry = { id, round, date, time, timeUtc, team1Id, team2Id, group,
                groundId, num, stage }
if (m.score1 !== undefined) { entry.score1 = m.score1; entry.score2 = m.score2 }
if (m.goals) entry.goals = m.goals
if (m.lineups) entry.lineups = m.lineups
if (m.bookings) entry.bookings = m.bookings
if (m.substitutions) entry.substitutions = m.substitutions

The deploy workflow (deploy.yml) adds a new step after enrichment:

- run: node pipeline/squads.mjs        # refresh squad data
- run: node pipeline/enrich-goals.mjs   # existing — adds scorerNo
- run: node pipeline/enrich-lineups.mjs # new — adds lineups/cards/subs
Order matters: The enrichment scripts must run after update.mjs (which regenerates matches.json from source). This is already the pattern for enrich-goals.mjs.

4. Frontend Display — MatchPage

Three new collapsible sections below the goal timeline in app/src/pages/index.tsx (MatchPage component). Each section only renders when data exists, so matches without enrichment show nothing new.

6. Implementation Order

  1. Extend the TypeScript interfaces in index.tsx and TeamDetail.tsx — add TeamLineup, BookingEvent, SubEvent to the Match interface.
  2. Update update.mjs cleanMatches — add pass-through for the three new fields so they survive pipeline regeneration.
  3. Build enrich-lineups.mjs — Wikipedia scraper:
    • Fetch article by convention URL
    • Parse "Details" section HTML for starting XI + subs
    • Parse "Statistics" section for aggregate card counts
    • Patch matches.json in-place (same pattern as enrich-goals.mjs)
  4. Update deploy.yml — add the enrichment step after enrich-goals.mjs.
  5. Build the MatchPage UI — add Starting XI, substitutions, and cards sections.
  6. Manual override file — create match-events-override.json for matches the scraper can't reach.
  7. (Future) FIFA PDF parser — per-player booking data + formation info for all matches.

7. Open Questions

  1. Data source priority: Start with the Wikipedia scraper (automated, partial coverage) or build the manual override system first (full control, manual effort)?
  2. Card granularity: Wikipedia only has aggregate counts per team, not per-player. Ok to start with aggregate card counts (🟨 2 - 🟨 3 per team) until FIFA PDF parsing is added?
  3. Formation: Only available in the FIFA tactical PDFs. Worth the PDF parsing effort, or skip it and just show a plain XI list without formation badges?
  4. Bench display: Show all bench players (up to 15) collapsed under a "Subs" line, or only show those who actually played (i.e. those mentioned in sub events)?
  5. Lang/i18n: Player names from Wikipedia are in English. The squad data has Chinese names keyed by shirt number — should the lineup display use the same EN→ZH translation logic as goalscorers?
