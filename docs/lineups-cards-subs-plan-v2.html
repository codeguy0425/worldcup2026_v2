Enhancement Proposal · Phase 1

Lineups, Cards & Substitutions WC2026 Redesign — Integration Strategy

A phased approach for displaying starting XIs, bookings, and substitutions on the match page — covering data sourcing, pipeline integration, frontend UI, and the design decisions that govern each layer.

Data Audit

1. Current Data Landscape

Three potential data sources exist, each with a different level of completeness. Understanding their coverage determines the implementation strategy.

Source Lineups Cards Subs Coverage
openfootball worldcup.json — — — 104 matches — goals only
Wikipedia match reports ✅ Aggregate ✅ 1 article exists; many more will appear after matches
FIFA tactical PDFs ✅ ✅ ✅ Published for every match — includes formation + per-player cards
Key finding. Only one match-report article exists on Wikipedia so far (Tunisia v Japan). The openfootball JSON — the pipeline's primary source — carries goals only. FIFA publishes a tactical-lineup PDF for every match, but the format requires a PDF extraction step.

Type Contract

2. Data Model

Three new interfaces extend the Match type. All fields sit inline in matches.json — the match page already fetches it, so no additional useJson calls are introduced. 

// ——— New interfaces added to index.tsx ———

interface TeamLineup {
  teamId: string
  formation?: string         // "4-3-3", "3-4-2-1"
  startingXI:  PlayerInLineup[]  // 11 starters
  substitutes: PlayerInLineup[]  // bench (up to 15)
}

interface PlayerInLineup {
  no:       number
  pos:      string   // GK / DF / MF / FW or specific (CB, CM, CF)
  name:     string
  captain?: boolean
}

interface BookingEvent {
  minute:        number
  stoppageTime?: number
  teamId:        string
  player:        string
  card:          'yellow' | 'red' | 'second-yellow'
}

interface SubEvent {
  minute:        number
  stoppageTime?: number
  teamId:        string
  off:  { no: number; name: string }
  on:   { no: number; name: string }
}

// ——— Updated Match interface ———

interface Match {
  // … existing fields …
  goals?:          GoalEvent[]
  lineups?:        TeamLineup[]    // new
  bookings?:       BookingEvent[]  // new
  substitutions?:  SubEvent[]     // new
}
Card granularity. Wikipedia provides per-team aggregate counts only (🟨 2 – 🟨 3). Per-player booking data requires FIFA's PDFs. The model supports both: per-player events for detailed display, and a fallback aggregate view for Wikipedia-sourced data.

Pipeline

3. Data Population — Three Paths

Path A

Wikipedia Scraper

pipeline/enrich-lineups.mjs — fetches each match's Wikipedia article, parses the Details section for XIs + subs, and the Statistics section for aggregate card counts.

3-4 days Partial coverage

Path B

FIFA PDF Parser

Downloads each tactical lineup PDF from fdp.fifa.org, extracts formation, per-player bookings, and full subs. Gold standard, but format may shift.

5-7 days Full coverage

Path C

Manual Override

match-events-override.json — hand-written JSON for matches no automated source covers. Same pattern as the existing scorer-no-override.json.

1 day Manual per match

Recommended start: Wikipedia scraper + manual override

Begin with the automated scraper covering matches that have articles. Use a manual override file for the remainder. The FIFA PDF parser can be added as a second phase when full coverage is needed. The Wikipedia article URL follows a predictable convention: 

https://en.wikipedia.org/wiki/{Team1}_v_{Team2}_({year}_FIFA_World_Cup)

Scraper flow

for each match M in matches:
  if M has no score → skip
  article = fetch(PAGE_URL(M.team1, M.team2))
  if article is 404 → continue

  // Parse "Details" section (index 9)
  details = article.getSection("Details")
  M.lineups = extractLineups(details, M.team1Id, M.team2Id)
  M.substitutions = extractSubs(details, M.team1Id, M.team2Id)

  // Parse "Statistics" section for card counts
  stats = article.getSection("Statistics")
  M.bookings = extractCards(stats, M.team1Id, M.team2Id)

  // Patch matches.json in-place
  patchFile(M)

Pipeline integration

The cleanMatches function in update.mjs gains three new pass-through fields: 

const entry = { id, round, date, time, timeUtc, team1Id, team2Id, group,
                groundId, num, stage }
if (m.score1 !== undefined) { entry.score1 = m.score1; entry.score2 = m.score2 }
if (m.goals) entry.goals = m.goals
if (m.lineups) entry.lineups = m.lineups
if (m.bookings) entry.bookings = m.bookings
if (m.substitutions) entry.substitutions = m.substitutions

Deploy workflow (deploy.yml) — the new step runs after goal enrichment:

- run: node pipeline/squads.mjs
- run: node pipeline/enrich-goals.mjs
- run: node pipeline/enrich-lineups.mjs  // new
Order constraint. Enrichment scripts must run after update.mjs, which regenerates matches.json from source. This follows the established pattern for enrich-goals.mjs.

Decisions

7. Open Questions

  1. Data source priority. Start with the Wikipedia scraper (automated, partial coverage) or build the manual override system first (full control, manual effort)?
  2. Card granularity. Wikipedia only has aggregate counts per team, not per-player. Acceptable to start with aggregate card counts (🟨 2 – 🟨 3 per team) until FIFA PDF parsing is added?
  3. Formation display. Only available in FIFA tactical PDFs. Worth the PDF parsing effort, or skip formation badges and show a plain XI list?
  4. Bench display. Show all bench players (up to 15) collapsed under a "Subs" line, or only show those who actually played (i.e., mentioned in sub events)?
  5. Chinese name translation. The squad data has Chinese names keyed by shirt number. Should the lineup display use the same EN → ZH translation logic as goalscorers?
